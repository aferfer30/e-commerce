# Part 2.1A — Database Architecture

# 1. Database Philosophy

## 1.1 Purpose

The database is the foundation of the entire application.

Every feature implemented within the website ultimately depends on the database.

A well-designed database reduces bugs, simplifies development, improves performance, and makes future expansion significantly easier.

The database should prioritize:

* Simplicity
* Reliability
* Consistency
* Maintainability
* Scalability

It should never be designed only to satisfy current requirements.

Instead, every table should be capable of supporting future features with minimal structural changes.

---

# 1.2 Design Principles

The database follows several fundamental principles.

## Single Source of Truth

Every piece of information should exist only once.

Example:

A product's price exists only inside the Product table.

The OrderItem stores a snapshot of that price at purchase time rather than referencing the current product price, ensuring historical accuracy.

---

## Normalization

The database should avoid unnecessary duplication.

Related information belongs in separate tables connected through relationships.

Example:

A category should not be stored as plain text inside every product.

Instead:

Product

↓

Category

This ensures consistency and simplifies updates.

---

## Controlled Redundancy

Some information may intentionally be duplicated when it preserves historical data.

Examples include:

Purchase price.

Shipping cost.

Discount amount.

Order total.

These values should never change after an order is completed, even if the original product data changes.

---

## Integrity

Every relationship should be valid.

An order cannot reference a product that does not exist.

A review cannot belong to a nonexistent customer.

A product image cannot belong to a nonexistent product.

Integrity is more important than convenience.

---

## Predictability

Every table should have a clear responsibility.

Avoid "miscellaneous" tables.

Avoid generic JSON blobs unless absolutely necessary.

Data should remain structured.

# Part 2.1B — Backend Services & Business Logic

# 18. Backend Philosophy

The backend is responsible for enforcing every business rule of the application.

It should never simply forward requests to the database.

Instead, every request must pass through validation, business logic, authorization (when required), and error handling before any database operation is performed.

The backend is the single source of truth for application behavior.

Even if the frontend performs validation, the backend must validate everything again.

Client-side validation improves user experience.

Server-side validation protects data integrity.

---

* Managing products.

* Managing orders.
* Managing customers.
* Managing wishlists.
* Managing shopping carts.

It should only return structured data.

# Part 2.2A — API Architecture & Communication

# 45. API Philosophy

The API acts as the communication layer between the frontend and the backend.

The frontend should never communicate directly with the database.

Every request must pass through the API.

The API is responsible for:

* Receiving requests.
* Validating data.
* Calling the correct service.
* Returning standardized responses.
* Preventing invalid operations.
* Protecting the database.

The API should remain predictable, consistent, and easy to understand.

---

# 46. API Design Principles

Every endpoint should follow these principles.

## Consistency

Endpoints should follow the same naming conventions throughout the project.

Developers should never have to guess where an endpoint belongs.

---

## Simplicity

Endpoints should perform one primary task.

Avoid endpoints responsible for multiple unrelated operations.

---

## Readability

Routes should describe resources rather than actions.

Good examples:

/api/products

/api/categories

/api/orders

/api/reviews

Avoid unnecessary verbs inside URLs.

---

## Statelessness

Each request should contain all required information.

The server should not depend on temporary request memory.

---

## Predictability

The same request should always produce the same result when the underlying data has not changed.

---

Example structure:

app/

    ├── orders/

/api/categories

/api/reviews

/api/users

Avoid abbreviations.

Avoid inconsistent naming.

Avoid mixing singular and plural names.

Never modify data.

Cards.

Typography.

---

# 120. Final Design Principles

Every design decision should satisfy the following questions:

Does it improve usability?

Does it improve readability?

Does it improve trust?

Does it improve product presentation?

Does it remain consistent with the rest of the website?

Would it feel appropriate for a professional Algerian technology retailer?

If the answer to any of these questions is "no," the design should be reconsidered before implementation.

The ultimate objective is to create an e-commerce website that feels immediately familiar to Algerian shoppers while standing noticeably above the visual quality, polish, and user experience of most local online stores. The website should encourage confidence, make product discovery enjoyable, and serve as a portfolio project that demonstrates both strong engineering practices and exceptional design execution.

# Part 3.4 — User Experience & Interaction Design

# 201. UX Philosophy

User experience is the foundation connecting all visual and technical decisions.

The website should not only look professional.

It should feel:

* Fast.
* Reliable.
* Clear.
* Comfortable.
* Trustworthy.
* Easy to use.

The goal is to create an experience where customers can discover products, make decisions, and complete purchases without unnecessary friction.

The website should respect the expectations of Algerian online shoppers while providing a more polished experience than typical local e-commerce websites.

---

# 202. UX Priorities

The experience should prioritize:

## Trust

Customers should feel confident buying from the store.

Trust is especially important in the Algerian market because many customers are cautious about online purchases.

Trust elements include:

* Clear prices in DA.
* Delivery information.
* Cash on Delivery availability.
* Warranty information.
* Customer reviews.
* Professional product presentation.
* Transparent checkout process.

---

## Speed

Users should not wait unnecessarily.

The website should:

* Load quickly.
* Respond instantly to interactions.
* Avoid heavy animations.
* Optimize images.
* Reduce unnecessary requests.

A beautiful website that feels slow provides a poor experience.

---

## Clarity

Every page should answer:

What is this?

Why should I care?

What can I do next?

Important information should always be visible.

---

## Simplicity

The user should complete tasks with minimal effort.

Examples:

Adding products to cart.

Finding a product.

Checking delivery options.

Completing checkout.
